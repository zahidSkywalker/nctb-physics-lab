The Builder’s Philosophy
Velocity is a feature. A working prototype today is worth more than a perfect system next
month. Ship something. Learn from it. Improve it.
But velocity without craft creates debt. Moving fast without thinking creates systems that
slow you down permanently. The fastest path to done is the one you don’t have to redo.
Design and code are the same decision. A button’s padding is a visual decision and a
code decision simultaneously. The best builders don’t hand off between “design” and “dev”
— they hold both.
The user doesn’t care how the code is structured. But the user absolutely cares if the
interface feels wrong, loads slowly, or breaks unexpectedly. Build for the person using it.
Read the room, then build for it. A startup MVP demands different decisions than a
production enterprise system. A personal portfolio demands different decisions than a SaaS
dashboard. Context determines craft level — but craft is never zero.
Part 1 — Vibe Coding
1.1 — What Vibe Coding Actually Is
Vibe coding is not sloppy coding. It is high-velocity creative development — the practice
of moving from idea to working product as fast as possible, using intuition, momentum, and
intelligent shortcuts without sacrificing the foundations that matter.
THE VIBE CODING MANIFESTO
═══════════════════════════
DONE > PERFECT (initially)

Ship something you can touch and improve.
SCAFFOLD FAST, BUILD CAREFULLY
Use every scaffold, generator, and template available.
They eliminate setup so you can focus on what's unique.
Understand what they generated — don't treat it as magic.
FEEL THE FLOW
When the code is writing itself, don't stop to over-engineer.
Ride the momentum. Clean up when momentum naturally pauses.
SHIP → LEARN → ITERATE
The real spec emerges from using the thing.
Version 1 is always wrong. Ship it so you can write version 2.
INSTINCT + VERIFICATION
Trust your instincts on architecture and design.
Verify with tests, type safety, and a working demo.
Never trust instincts on security or data integrity — verify those always.

PHASE 1 — CONCEPT (5 minutes)
What is the ONE thing this must do?
Who uses it?
What does "working" look like?
Answer these before touching the keyboard.
Vercel            — Next.js native, best DX, edge functions, analytics

Cloudflare Pages  — edge-first, Workers, D1, R2, KV — very fast
Coolify           — self-hosted PaaS, deploy anything, your server
PHASE 2 — SCAFFOLD (5-15 minutes)
Pick the right stack for the job (not the newest stack)
Run the scaffolding command
Install core dependencies
Set up environment variables
Verify: does it run?
PHASE 3 — SKELETON (30-60 minutes)
Build the basic structure — routing, layout, main components
No data, no backend, no styling yet
Get the bones right before adding muscle
Verify: can you navigate the entire app?
PHASE 4 — CORE FUNCTIONALITY (hours to days)
Build the thing the app actually does
Start with the happy path
No error handling yet — that comes next
Verify: does the main use case work end-to-end?
PHASE 5 — HARDEN (hours)
Add error handling at every failure point
Add loading states
Handle edge cases you know will happen
Add form validation
Verify: does it survive bad input and slow connections?
PHASE 6 — POLISH (hours)
Styling and animations
Responsive behavior
Accessibility basics
Performance optimization
Verify: would you be comfortable showing this to someone?
PHASE 7 — SHIP
Deploy
Set up monitoring
Document the essentials
Then: iterate based on what you learn

function useDebounce<T>(value: T, delay: number): T {
  const [debouncedValue, setDebouncedValue] = useState(value)
  useEffect(() => {
    const handler = setTimeout(() => setDebouncedValue(value), delay)
    return () => clearTimeout(handler)
  }, [value, delay])
  return debouncedValue
}
